# AI-900 Practice A — Summary Bullet Points

1. **PaaS** lets you develop and deploy apps without managing underlying infrastructure (VMs, OS, etc.)
2. **On-demand** cloud computing means you can quickly provision and release resources based on actual demand — no need to buy servers.
3. **Data & access** is the customer's responsibility across all service models (IaaS, PaaS, SaaS) in the Azure Shared Responsibility Model.
4. **Hybrid cloud** combines on-premises infrastructure with public cloud (e.g., Azure), enabling scenarios like cloud bursting for elastic capacity while keeping regulated workloads on-prem.
5. **Hybrid cloud** is the go-to model when compliance requires sensitive data on-prem but you still need public cloud elasticity for traffic spikes.
6. **Consumption-based model** (pay-as-you-go): metered usage, scale with demand, stop when idle — no upfront CapEx, no fixed fees, costs track actual usage.
7. **Reserved instances** (1–3 year commitment) give high discounts with no eviction risk — best for steady, predictable 24/7 workloads. Spot VMs are cheaper but can be evicted.
8. **Azure Functions** (serverless) is ideal for event-driven, unpredictable workloads with no server management and per-execution billing (e.g., process images on blob upload).
9. **Scalability** is the ability to add/remove instances to match demand (e.g., 10x traffic surge). Resiliency = surviving failures; Durability = data not getting lost.
10. **Azure Policy** enforces governance at scale (allowed locations, required tags). RBAC controls *who* can act; Policy controls *what* is allowed.
11. **Azure Service Health** + published reliability guidance (SLAs, maintenance docs) improve predictability for operations and support reliability planning.
12. **Azure Policy** with the built-in "Allowed Locations" (Deny effect) blocks noncompliant deployments to unapproved regions — centralized, auditable governance at scale.
13. **Azure portal** provides a centralized web-based interface for managing all Azure resources across subscriptions. **ARM templates** enable Infrastructure as Code for consistent, repeatable deployments. **Azure Automation** automates operational tasks (start/stop VMs, run scripts, schedule maintenance) to reduce manual effort and improve consistency.
14. **IaaS** (Infrastructure as a Service) gives customer control over guest OS patching, runtime/middleware, and installed software while provider manages physical hardware — ideal for legacy apps requiring custom OS-level settings and specific patching schedules during lift-and-shift migrations.
15. **PaaS** (Platform as a Service) lets developers deploy custom code quickly without patching OS or managing web runtime — provider handles OS, runtime, middleware, and infrastructure while customer controls app code, settings, and deployments.
16. **SaaS** (Software as a Service) means provider manages app + platform + hardware while customer only manages users + data — ideal when vendor handles all application updates and platform maintenance while customer focuses on user access and data entry (e.g., ticketing systems).
17. **Azure Private Link** (private endpoint) assigns a private IP from your VNet to a PaaS service, fully eliminating public internet exposure and public endpoints — unlike Service Endpoints which still use the service's public IP address.
18. **PaaS** (Platform as a Service) abstracts and manages the underlying guest OS — you CANNOT get admin access to install custom agents/drivers or perform OS-level customizations. Use **IaaS** (Azure Virtual Machines) when full OS control is required, even though it involves more operational overhead.
19. **Availability Zones** provide high availability within a single Azure region using physically separate datacenters with independent power, cooling, and networking, connected via high-bandwidth low-latency links — protects against datacenter-level failures without requiring cross-region failover.
20. **Datacenter** is the physical building/facility that directly houses Azure's compute, storage, and networking hardware — availability zones (grouping one or more datacenters) and regions (grouping zones/geographic areas) are logical constructs, not the physical location itself.
21. **Azure Monitor** is the central service that collects telemetry data (metrics, logs, activity logs) from Azure resources and hybrid/on-premises environments, analyzes it through dashboards, queries, and alerts, and enables actions like automated responses via alerts and autoscale.
22. **Azure subscriptions** provide isolated billing (separate invoices), independent quota management, and resource limits while still allowing central governance through placement under a common management group for policies, RBAC, and cost management at scale.
23. **Management groups** allow organizing multiple subscriptions into a hierarchy and applying Azure Policy definitions, policy assignments, and RBAC role assignments at the management group level once — governance controls automatically inherit to all child subscriptions and resource groups for consistent, scalable governance.
24. **Azure hierarchy**: Management groups contain subscriptions, which contain resource groups, which contain resources — governance flows downward from management groups to child subscriptions and resource groups.
25. **Compute types**: Virtual machines (VMs) = full OS control; Containers = packaged microservices with consistent runtime; Functions = event-driven serverless code execution.
26. **Network interface** (NIC) connects a VM to a subnet and provides IP configuration (private/public IP) — required for VM network connectivity.
27. **Containers** package the application and dependencies into a portable image for consistent deployment across environments — build once, run anywhere without OS management.
28. **Azure Cloud Shell** is a browser-based interactive CLI environment (PowerShell or Azure CLI) with pre-installed tools — no local installation required.
29. **Managed identities** give Azure workloads an Entra ID identity to request tokens for accessing services — no secrets to manage or rotate, eliminating credential leakage risk.
30. **VPN Gateway** provides encrypted tunnels over the public internet between on-premises networks and Azure VNets — secure site-to-site or point-to-site connectivity without dedicated private circuits.
31. **Azure Storage types**: Blob Storage = unstructured objects/media; Azure Files = SMB-compatible file shares; Queue Storage = simple message decoupling; Table Storage = key-value structured data.
32. **Guest users** are external identities invited to collaborate on Azure resources with limited, scoped access — partner users from other organizations sign in with their own org accounts.
33. **Conditional Access** (Entra ID) evaluates sign-in context (device, location, risk) and applies "if this, then require" policy decisions — require extra verification for unmanaged devices or outside corporate network.
34. **Zero Trust** security model = verify explicitly, use least privilege, assume breach — network location is NOT a trusted signal, continuous verification required regardless of request origin.
35. **Defense in depth** = multiple security layers across identity, network, application, and data — if one control fails, others reduce risk (not single-layer perimeter protection).
36. **RBAC** (Role-Based Access Control) components: role assignments (connect roles to principals at a scope), built-in roles (predefined permissions like Owner/Contributor/Reader), and scope inheritance (permissions flow down hierarchy from management groups to subscriptions to resource groups).
37. **Microsoft Defender for Cloud** provides security posture management, recommendations, secure score tracking, and threat protection across subscriptions — single place for security alerts and best-practice guidance.
38. **Region** is the most direct factor affecting Azure service pricing — identical configurations in different regions can have different unit rates due to local costs, demand, and currency fluctuations.
39. **Tags** are key-value metadata labels for cost allocation, filtering, and organization — enable billing by tag, resource grouping, and policy enforcement without affecting resource runtime behavior.
40. **Azure subscription** is a billing container that organizes resources, tracks usage, defines access boundaries, and provides quota limits — the fundamental unit for cost management and access control.
41. **Storage access tiers**: Hot tier = frequent access with immediate retrieval; Cool tier = lower cost for infrequent access with immediate retrieval; Archive tier = lowest cost but hours-long retrieval latency.
42. **Zone-redundant storage (ZRS)** replicates data across availability zones within the same region — protects against datacenter failures while keeping data within the region for compliance (unlike GRS which is cross-region).
43. **General-purpose storage accounts** support all Azure Storage services (Blob, Files, Queue, Table) under a single account — default choice for scenarios requiring multiple storage types.
44. **AzCopy** is a command-line tool for high-performance data transfer to/from Azure Storage — optimized for large-scale transfers, resumable operations, and parallel processing.
45. **Microsoft Purview** provides data governance capabilities: data catalog (discover and classify assets), data classification (label sensitive data), and data lineage (track data flow across systems) — not for resource deployment, RBAC, or networking.
46. **Azure Data Box** ships physical devices for offline bulk data transfer when network bandwidth is insufficient — ideal for migrating terabytes to petabytes of data where online transfer would take too long.
47. **Entra ID** (formerly Azure AD) is the cloud identity and access management platform for user authentication, SSO, and application access. **Entra Domain Services** provides managed domain join without running domain controller VMs.
48. **Passwordless authentication** eliminates passwords entirely as a sign-in factor — uses Windows Hello for Business, FIDO2 security keys, or Microsoft Authenticator for stronger, phishing-resistant authentication.
49. **Cloud computing benefits**: global reach and scalability (deploy resources worldwide, scale dynamically), not increased upfront costs, zero security risk, or eliminating all software updates.
50. **Cloud deployment models**: Public cloud = shared infrastructure owned by third-party provider (multiple orgs share resources); Private cloud = dedicated infrastructure for one organization (ideal for sensitive data); Hybrid cloud = combines on-premises and public cloud.
51. **Shared responsibility**: Customer is responsible for application security, data, access control — Azure manages physical security, network infrastructure, and hardware maintenance.
52. **Governance and security benefits**: protect resources and ensure compliance, not automatically cut costs, eliminate backup needs, or remove all human errors.
53. **Resource locks**: Delete lock prevents accidental deletion while allowing configuration changes; Read-only lock blocks all changes including configuration updates (too restrictive for normal operations).
54. **Azure portal** is a browser-based graphical UI for creating, configuring, and monitoring Azure resources — no local installation required, ideal for visual management and discovery.
55. **Infrastructure as Code (IaC)** characteristics: version control (track changes in git), repeatable deployments (consistent environments), peer review (code reviews before applying) — NOT manual changes, one-time setup, or untracked edits.
56. **Azure Advisor** provides automated best-practice recommendations across cost, performance, reliability, and security — analyzes your environment and suggests improvements without automatically changing resources.
57. **Resource groups** organize related resources for lifecycle management (deploy, update, delete together), access control (RBAC), and policy application — NOT for extra storage, automated deployment, or network monitoring.
58. **Region pairs** are Microsoft-defined relationships between two regions in the same geography for coordinated disaster recovery — service updates roll out to paired region first, ensuring one region is always available as failover target.
59. **Azure Virtual Desktop (AVD)** delivers managed virtual desktops to end users at scale — centrally managed, scalable, ideal for temporary desktops (students, contractors) or remote work scenarios.
60. **Multifactor authentication (MFA)** requires two or more verification factors (something you know, something you have, something you are) to increase security beyond just passwords.
61. **Virtual machines** are individual Azure resources — the fundamental unit managed in Azure (not higher-level constructs like subscriptions or management groups).
62. **Azure Migrate** provides a centralized hub for discovering, assessing, and planning server migrations to Azure — assesses on-premises servers for Azure readiness, cost estimates, and migration recommendations.
